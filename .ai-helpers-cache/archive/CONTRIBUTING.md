## Canonical Sources & 2025 Best Practices for Code/Doc Sync

- **Canonical Sources:**
  - All documentation and LLM/agent introspection must reference the following as canonical sources:
    - `docs/docManifest.json` (doc manifest)
    - `.nootropic-cache/describe-registry.json` (describe registry)
- **Describe Registry & Doc Sync Enforcement:**
  - All modules, agents, adapters, plugins, and utilities (including all subfolders) must export a complete `describe()` and will be automatically included in the registry if present. All must be present in the doc manifest.
  - Run `pnpm run validate-describe-registry` and `pnpm run docs:check-sync` to validate code/doc sync and describe registry completeness. The registry aggregation is now fully dynamic and recursive.

  - See `README.md` and this file for workflow details and troubleshooting.

## 🚀 Automated Docs-as-Code & LLM/AI Discoverability (2025 Best Practice)

- The doc manifest (`docs/docManifest.json`) is **auto-generated** from the describe registry by [`scripts/generateDocManifest.ts`].
- Only implemented modules are required for code/doc sync; planned/in-progress features are tracked for roadmap/backlog automation and do **not** cause sync failures.
- The sync script ([`scripts/docs-check-sync.ts`]) enforces this and prints planned features for roadmap awareness.
- **Contributors should not manually edit the manifest**; always run the generateDocManifest script before checking sync.
- This ensures **zero-drift, LLM/AI-friendly, and future-proof documentation and discoverability**.
- All new features must export a `describe()` and will be auto-discovered and included in the registry and manifest.

## Onboarding Checklist: Doc/Code Sync & Canonical Sources (2025 Best Practices)

- For every new module, agent, or plugin:
  - Export a complete `describe()` (with name, description, usage, schemas, references). All new agents must be registry/describe/event-schema compliant, export LLM/AI-friendly describe() outputs, include advanced feature stubs and extension points, and be documented in docs/capabilities with best practices, usage, and troubleshooting. CI/CD and onboarding enforce compliance.
  - **Do not manually edit docs/docManifest.json.**
  - Run:
    ```sh
    pnpm tsx scripts/generateDocManifest.ts
    pnpm run docs:check-sync
    pnpm tsx scripts/generateOnboardingChecklist.ts
    pnpm tsx scripts/generateRecentMessages.ts
    pnpm tsx scripts/generateHealthStatus.ts
    ```
  - Add the module/capability to the backlog if planned/in-progress.
  - **Reference all automation scripts in onboarding and docs, including pluginMarketplaceCli.ts (plugin marketplace, rating, review) and submitPluginFeedback.ts (feedback automation).**
- Before submitting a PR:
  - Run `pnpm run validate-describe-registry` to check all modules export a valid `describe()`.
  - Run `pnpm run docs:check-sync` to enforce code/doc parity between `.nootropic-cache/describe-registry.json` and `docs/docManifest.json`.
  - CI/CD will block PRs if any described module is missing from docs, or any documented section is missing from code.
- Troubleshooting:
  - If code/doc sync fails, update both the `describe()` output and `docs/docManifest.json`. All new modules must export a `describe()` and will be automatically included in the registry.
  - See the error output for missing or duplicate entries.
  - Reference: `.nootropic-cache/describe-registry.json` and `docs/docManifest.json` are the canonical sources for LLM/agent introspection and documentation automation. The registry is always up to date and validated in CI.
  - See README.md for workflow details and troubleshooting.

## Advanced Context APIs (2024+)

The following advanced context APIs are available for all agents, plugins, and orchestration workflows:

- **Delta Snapshot:** `GET /context/delta` — Returns a delta snapshot for efficient incremental context handoff.
- **Semantic Index:** `GET /context/semantic-index` — Returns the semantic index for context-aware orchestration and LLM workflows.
- **Telemetry:** `GET /context/telemetry` — Returns telemetry events for advanced orchestration and observability.
- **Optimized Handover:** `POST /context/optimized-handover` — Returns an optimized, token-aware handover payload for agent context. Accepts `{ contextArr, agentConfig, modelTokenLimit }` in the request body.

### LLM/AI Usage Hints
- Use delta snapshots for efficient, incremental updates in long-running agent workflows.
- Leverage the semantic index for context-aware search, triage, and orchestration.
- Use telemetry events for observability, debugging, and compliance automation.
- Always use the optimized handover API for token-limited LLM handoff and agent-to-agent communication.

See also: `contextSnapshotHelper.ts`, `contextMutationEngine.ts`, `contextRealtimeServer.ts` for implementation details and extension points.

## 🧩 Plugin/Tool Safety: Zod Validation, Type Guards & Hot-Reload

- All plugin/tool inputs and outputs must be validated using Zod schemas.
- Use type guards at all plugin boundaries to ensure type safety.
- Reference input/output schemas in `describe()` for discoverability and LLM/agent introspection.
- The plugin registry enforces this pattern and will skip plugins that do not conform (see debug logs).
- **Dynamic event subscription:** Use `appContext.subscribeToEvent` (or `pluginManager.subscribe`) with the plugin name for tracking. All event subscriptions are automatically cleaned up on unload/hot-reload, preventing stale handlers and enabling robust hot-reload. See the updated example plugin and PluginManager for details.
- Contributors must follow this pattern for all new plugins/tools.

## 🚀 Distributed Event Bus: Kafka & NATS Integration

- All distributed agent, plugin, and event communication must use the `KafkaEventBus` or `NatsEventBus` abstraction (`adapters/KafkaEventBus.ts` or `memoryLaneHelper.ts`).
- All events must be validated at runtime using Zod schemas for type safety and schema governance.
- OTel traces/metrics/logs must be emitted for all event bus operations (see `otel.config.ts`).
- Contributors should follow orchestrator-worker, hierarchical, blackboard, and market-based multi-agent patterns as needed ([Confluent](https://www.confluent.io/blog/event-driven-multi-agent-systems/)).
- Plugins must support hot-reload and dynamic topic subscription.
- For NATS: Set `EVENT_BUS_BACKEND=nats` and configure `NATS_URL`, `NATS_STREAM`, `NATS_SUBJECT`, and `NATS_DLQ_SUBJECT`. Uses `nats.js` with JetStream for high-performance, subject-based routing, replay, DLQ, and OTel tracing. All events are validated at runtime using Zod schemas. See `memoryLaneHelper.ts` for implementation details.

See also: [Kai Waehner on Agentic AI](https://www.kai-waehner.de/blog/2025/04/14/how-apache-kafka-and-flink-power-event-driven-agentic-ai-in-real-time/), [Medium: Multi-Agent Orchestration](https://medium.com/@seanfalconer/how-to-build-a-multi-agent-orchestrator-using-flink-and-kafka-4ee079351161)

## 🧩 Registry/LLM/AI Compliance: Required Export Pattern

All agents, utilities, plugins, and adapters **must** default-export a plain capability object with the following properties:
- `name`: The capability name (string)
- `describe`: A function returning a docs-first, AI/LLM-friendly metadata object (see below)
- `schema`: A Zod schema (or JSON schema) describing the input/output for LLM/agent validation and introspection

**Example pattern:**
```ts
import { z } from 'zod';

export class MyUtility {
  static schema = z.object({ input: z.string() });
  static describe() {
    return {
      name: 'MyUtility',
      description: 'Does something useful.',
      methods: [/* ... */],
      schema: MyUtility.schema
    };
  }
}

const MyUtilityCapability = {
  name: 'MyUtility',
  describe: MyUtility.describe,
  schema: MyUtility.schema
};

export default MyUtilityCapability;
```

- All new modules following this pattern are **automatically included** in the registry, CLI, and HTTP API.
- The registry is always up to date and validated in CI.
- Zod schemas are used for runtime validation and LLM/agent introspection.
- See the "Registry/LLM/AI Compliance" section in the README for more details.

## Type Safety and Event Payload Validation (2025 Best Practices)

- All agent event payloads and task inputs **must** use `Record<string, unknown>` and type guards instead of `any` for extraction and validation.
